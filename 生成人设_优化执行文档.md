# 生成人设优化执行文档

## 1. 结论

`生成人设`这一步的目标，不是补一份漂亮的人物小传，而是把已选脑洞、导语和故事架构，转译成后续剧情生成可以稳定调用的`核心演员行动规则`。

它应该回答三个问题：

1. 谁必须出场，谁可以合并或删除。
2. 每个人在压力下会怎么行动，怎么说话，怎么把冲突推高。
3. 这些人物如何服务第1-4章执行蓝图、第4章末付费点、第5-10章后续反转。

因此，优化方向是：从当前的`3-5个核心演员分配表`升级为`3-5个核心演员冲突驱动表`。保留轻量字段，但每个字段都必须能被后续剧情点、选项、钩子直接引用。

## 3. 当前问题

当前项目的`生成人设`实现位于：

- `src/features/architectureSetup/architectureEngine.js`
- `src/features/architectureSetup/architectureState.js`
- `src/features/architectureSetup/ArchitectureSetupStage.vue`

当前演员字段为：

- `name`
- `persona`
- `voice`
- `reaction`
- `emotion`
- `conflict`

这个结构的优点是轻、好展示、不会过度膨胀。问题是字段仍偏抽象：

- `persona`容易变成人格标签，不能保证会影响剧情。
- `voice`和`reaction`可以塑造人物，但不一定绑定主线。
- `emotion`和`conflict`容易重复，缺少明确的行为升级链。
- 反派、伙伴、工具人之间的功能差异不够硬，后续剧情可能随机加人。
- 缺少“第几章承担什么人物任务”的约束，导致人设和10章架构可能脱节。

## 4. 优化目标

### 4.1 思路清晰

生成人设只做一件事：建立`核心演员如何推动故事`。

它不重复：

- 不重复导语正文。
- 不重复10章事件链。
- 不写完整人物生平。
- 不扩写正文。
- 不新增世界观解释。

### 4.2 有收益

优化后应带来四个收益：

1. 后续剧情点更稳：角色行为不会忽然变成另一个人。
2. 选项更有差异：不同选择能来自不同人物的行动策略。
3. 钩子更强：第4章末付费点能绑定人物秘密、关系翻面或行动代价。
4. 改稿更容易：用户能直接编辑某个角色的动机、边界或升级方式，而不是重写整份大纲。

### 4.3 不冗余

每个核心演员只保留能被剧情调用的信息。建议单人字段控制在8项以内，每项1-2句。

## 5. 推荐输出结构

建议把`actors`从当前6字段扩为8字段：

```json
{
  "actors": [
    {
      "name": "角色名",
      "role": "主角/对手/伤害者/伙伴/信使/见证者",
      "persona": "一句话人设定位",
      "goal": "他想得到、保住、逃离或证明什么",
      "pressureBehavior": "压力升高时他会做什么，绝不做什么",
      "voice": "说话味道，最好带一条可模仿的台词规则",
      "emotionFunction": "他主要制造共情、愤怒、酸涩、爽感、悬疑或反差中的哪一种",
      "plotFunction": "他在第1-4章、第4章末付费点或后段反转中承担什么任务"
    }
  ]
}
```

字段说明：

| 字段 | 用途 | 不要写成 |
|---|---|---|
| `name` | 保证后续引用稳定 | 花哨称号堆叠 |
| `role` | 明确人物功能，防止乱加角色 | 性格标签 |
| `persona` | 一句话抓住反差或核心设定 | 长篇履历 |
| `goal` | 提供行动方向 | “想幸福”“想复仇”这类空话 |
| `pressureBehavior` | 约束极端情况下的选择 | 泛泛“坚强勇敢” |
| `voice` | 让正文对话有辨识度 | 只写“冷淡”“温柔” |
| `emotionFunction` | 明确读者情绪收益 | 多种情绪全都要 |
| `plotFunction` | 绑定架构，避免人设孤立 | 重复10章大纲 |

## 6. 生成流程

### 第一步：读架构，确定人物名额

输入必须包含：

- 已选脑洞
- 导语
- 故事起点
- 核心冲突
- 10章事件链
- 前四章执行蓝图
- 第4章末付费点
- 付费后立即回报
- 后段大反转
- 反转铺垫

先判断需要几个核心演员：

- 必须有主角。
- 必须有主对手或主要伤害者。
- 若赛道是追妻/虐恋，通常需要伤害者、竞争者或误导源。
- 若是世情/爽文，通常需要反派、小boss、见证者或规则代表。
- 伙伴、工具人、路人只有在改变信息、资源、关系或公开局势时才进入核心演员。

### 第二步：按功能分配角色

每个角色先定功能，再定性格。

推荐功能优先级：

1. 主角：目标、受限理由、反击能力。
2. 主对手：资源优势、压迫方式、最终漏洞。
3. 伤害者/误判者：为何偏帮、为何看不见、为何后悔或被清算。
4. 作恶推动者：负责把仇恨值具体化、升级化。
5. 伙伴/信使/见证者：提供线索、资源、公开认证或情绪对照。

### 第三步：为每个人写行为规则

不要只写“她很清醒”“他很恶毒”。要写：

- 第一次被冒犯时，他怎么反应。
- 局势升级时，他会用什么资源。
- 他最怕什么被揭开。
- 他绝不会做什么，否则会OOC。
- 他的一次选择如何改变下一章局势。

### 第四步：绑定章节任务

每个核心演员至少绑定一个章节任务：

- 第1章：出场、压迫、误判、立目标。
- 第2章：试探、加码、资源差距显露。
- 第3章：证据、关系裂缝、退路压缩。
- 第4章：人物选择逼近付费点。
- 第5章后：付费回报、反转推进、清算或情感落点。

## 10. 后续剧情调用建议

当前 `buildArchitectureSummary()` 只摘要演员的`name/persona/conflict`。优化后应把演员摘要改成更可调用的格式：

```text
【核心演员】
1. 角色名｜功能｜目标｜压力行为｜剧情任务
```

这样后续这些环节都能吃到人设约束：

- 第一个剧情点
- 下一个剧情点
- 剧情选项
- 章节钩子
- 大钩子
- 最终成文

## 11. 验收标准

一次合格的`生成人设`结果应满足：

- [ ] 核心演员数量为3-5个。
- [ ] 每个角色删掉都会影响主线。
- [ ] 主角有明确目标、受限理由和行动能力。
- [ ] 对手有资源优势、升级方式和最终漏洞。
- [ ] 每个角色都有压力下的行为规则。
- [ ] 每个角色都有可辨识的说话味道。
- [ ] 每个角色明确制造一种主要情绪。
- [ ] 每个角色绑定至少一个剧情任务。
- [ ] 没有外貌堆砌、完整履历、世界观复述。
- [ ] 输出不会重复导语和10章大纲。

## 12. 最小实施顺序

建议按这个顺序落地，避免一次改太大：

1. 先改 `architectureEngine.js` 的 persona 提示词和 JSON schema。
2. 再改 `architectureState.js` 的 actor 字段与验证。
3. 再改 `ArchitectureSetupStage.vue` 的表单字段。
4. 最后改 `buildArchitectureSummary()`，让后续剧情生成读到新字段。
5. 跑 `npm run build` 验证。

## 13. 不建议做的事

- 不建议把人设扩成几十个字段；字段越多，后续越难稳定使用。
- 不建议每个角色都写完整外貌、童年、MBTI、口头禅、人物弧光。
- 不建议把`生成人设`做成第二个`生成架构`。
- 不建议让AI在这一步继续生成章节细纲或正文。
- 不建议输出无功能配角；短篇里无功能角色会稀释冲突。
