# 【生成导语】优化执行文档

## 1. 结论

“生成导语”这一步不应该只是把脑洞改写成一段好看的开头。

更好的定位是：把已选脑洞压缩成一个可直接展示给读者的开篇引子，同时为后续“生成架构、人物、第一个剧情点”锁定核心矛盾、主角反应、情绪方向和继续阅读钩子。

一句话定义：

> 导语 = 脑洞落地后的第一段销售文案 + 第一场情绪冲突 + 后续主线的起点。

这样做有三个收益：

1. 读者收益：前 10 秒知道“这故事有什么冲突、为什么想看下去”。
2. 创作收益：后续架构不容易跑偏，因为主角目标和核心矛盾已经被导语钉住。
3. 工具收益：导语不再和脑洞、架构、第一剧情点重复，而是承担独立的“吸引 + 定调 + 承接”功能。

## 2. 当前问题

当前提示词已经要求 150-250 字、第一人称、短句分行、悬念、开放结尾、对话比例。

问题不在格式，而在任务目标偏模糊：

1. “吸引读者”太泛，模型容易写成普通简介或情绪化短段。
2. 没有强制导语包含“核心矛盾、仇恨、人设、钩子”这几个高价值骨架。
3. “承接脑洞”有要求，但没有要求导语产出的冲突必须成为后续主线起点。
4. 对话比例要求可能导致模型硬塞对话，反而稀释导语最该完成的冲突压缩。
5. 导语和第一剧情点边界不够清晰：导语应该抛出强冲突和期待，不应该开始铺完整剧情。

## 3. 这一步应该做什么

### 3.1 必须完成的三件事

第一，建立读者即时兴趣。

导语第一屏必须出现抓眼球的事件、关系冲突或异常背景。不要从天气、心情、童年、世界观解释开始。

第二，建立情绪账。

读者必须知道谁让“我”痛、怒、委屈、震惊或不甘。短篇导语的重点不是信息完整，而是让读者立刻站队。

第三，建立继续阅读钩子。

结尾要停在“事情刚要反转/主角刚要行动/真相露出一角”的位置。不要把复仇、打脸、真相、结局一次讲完。

### 3.2 必须包含的五个隐性模块

导语最终只输出正文，不加标签。但提示词内部必须要求模型完成以下五个模块：

1. 背景：如果脑洞有特殊规则、重生、系统、倒计时、身份错位等设定，第一时间交代必要信息。
2. 核心矛盾：人和人的冲突，或主角被迫面对的两难处境。
3. 仇恨/痛点：反派或环境对主角造成的伤害，负责拉动读者情绪。
4. 人设反应：主角面对冲突时的第一反应，决定爽文、虐文、疯感、隐忍或反击基调。
5. 钩子：情绪或情节反转，负责把读者推向下一步。

推荐顺序：

> 背景/矛盾开场 -> 仇恨加压 -> 主角反应 -> 钩子收尾

如果没有特殊背景，则直接从核心矛盾开场。

## 4. 不应该做什么

生成导语不要承担这些任务：

1. 不要重新发散脑洞。脑洞已经选定，导语只负责落地。
2. 不要生成完整架构。最多暗示主线，不要列章节、不写起承转合。
3. 不要生成人设卡。只能通过行动、语言、选择体现人设。
4. 不要写第一章完整剧情点。导语停在冲突爆发前后，不负责推进一个完整场景闭环。
5. 不要解释设定。只交代读者理解冲突所必需的信息。
6. 不要把后续爽点兑现完。导语负责制造期待，不负责付清期待。

## 5. 推荐输出标准

### 5.1 正文标准

建议默认：

1. 字数：180-260 字。
2. 人称：第一人称。
3. 排版：一句一行。
4. 开场：前 30 字内出现冲突、异常信息或强关系张力。
5. 情绪：至少有一个明确痛点或仇恨来源。
6. 主角：必须出现一次能体现主角性格的选择、反应或台词。
7. 结尾：用反问、反击、异常发现、重生节点、倒计时、门被敲响、对方一句狠话等方式卡住。

### 5.2 对话标准

不要机械要求 40%-50% 对话。

更好的规则是：

1. 至少有一句高信息量对话。
2. 对话必须承担冲突、羞辱、威胁、误解、反击或信息差。
3. 禁止寒暄、解释型对话、无效铺垫。
4. 如果脑洞更适合强叙述开场，可以降低对话比例，但不能降低冲突密度。

### 5.3 质量验收

导语生成后，用以下 8 条检查：

1. 第一眼能否看出卖点？
2. 是否有明确冲突对象？
3. 是否让读者产生站队情绪？
4. 主角是否有鲜明反应，而不是被动介绍？
5. 结尾是否留下“下一步必须看”的问题？
6. 是否没有偷跑到完整复仇、完整真相或结局？
7. 是否能自然接到“生成架构”？
8. 是否能自然扩成第一章第一个剧情点？

少于 6 条通过，则应重生成或局部重写。

## 6. 提示词优化建议

### 6.1 默认提示词建议

把当前“导语是放在小说开头的引子/简介”改成更具体的任务定义：

```text
你是一位短篇小说开篇策划和正文作者。请基于已选脑洞，写一段可直接展示给读者的导语。

导语不是故事简介，也不是大纲。
它要在 180-260 字内完成四件事：
1. 用核心矛盾或特殊背景开场，让读者立刻知道故事卖点。
2. 写出反派、关系或处境对“我”造成的伤害，快速建立情绪账。
3. 通过“我”的一句话、一个动作或一个选择，立住主角人设和故事基调。
4. 结尾停在情绪反转、行动前夜、真相露出一角或危险逼近的位置，让读者想继续看。

必须承接脑洞，不要另起设定。
不要生成章节大纲、人物卡或完整第一章。
不要把复仇、真相、反转或结局提前兑现。
```

### 6.2 运行时硬性补充建议

把当前导语 runtime guard 扩展为：

```text
【导语生成硬性补充要求】
1. 导语必须承接已选脑洞，只能把脑洞落地为开篇，不得新增一套主线。
2. 前 30 字内必须出现核心矛盾、异常背景、羞辱、危险或强信息差之一。
3. 正文中必须隐含完成：核心矛盾、仇恨/痛点、主角反应、结尾钩子。
4. 结尾必须保留开放性，停在读者想继续看下去的位置，不得提前解决冲突。
5. 对话只保留高信息量冲突对话；不要为了比例硬塞寒暄。
6. 不要输出标签、分析、标题或代码块，只输出导语正文。
```

## 7. 和后续步骤的边界

### 7.1 对“生成架构”的价值

导语应该给架构提供四个明确输入：

1. 故事起点：第一个冲突场景是什么。
2. 核心矛盾：主角和谁/什么机制对抗。
3. 情绪方向：爽、虐、疯、悬疑、反击、隐忍等。
4. 长钩子：后续 4 章或 10 章要兑现什么期待。

### 7.2 对“生成人设”的价值

导语不直接写人设卡，但要通过主角反应暴露人设。

例如：

1. 主角忍下羞辱，说明后续可能走隐忍蓄力。
2. 主角当场反击，说明后续可能走高爽打脸。
3. 主角看似顺从但暗藏计划，说明后续适合信息差和反转。

### 7.3 对“第一个剧情点”的价值

第一个剧情点应该承接导语最后的钩子，扩写成一个完整可选节点。

因此导语结尾不要写死：

1. 不要已经惩罚完反派。
2. 不要已经揭开最终真相。
3. 不要已经完成主角目标。
4. 不要跳到几天后、几年后，导致第一剧情点无法承接。

## 8. 推荐执行方案

### 阶段一：只改提示词，不改数据结构

优先调整 `guide` 默认提示词和 runtime guard。

原因：当前系统已经支持提示词配置和本次额外要求，改提示词收益高、风险低。

验收方式：

1. 选 3 个不同脑洞生成导语。
2. 用第 5 节的 8 条标准评分。
3. 检查能否顺畅生成架构和第一剧情点。

### 阶段二：增加“导语质量自检”但不展示给用户

如果后续要进一步稳定质量，可以让模型先内部检查五个模块，再只输出正文。

不建议一开始把结构标签显示在界面里。用户当前需要的是可读正文，结构标签容易增加噪音。

### 阶段三：需要时再做结构化导语资产

如果以后要把导语沉淀进 DNA 资产库，可以额外保存结构化字段：

```json
{
  "intro_text": "导语正文",
  "core_conflict": "核心矛盾",
  "emotion_trigger": "第一情绪",
  "protagonist_reaction": "主角反应",
  "hook_mechanism": "结尾钩子机制",
  "mainline_seed": "后续主线起点"
}
```

但这属于后续增强，不是当前第一优先级。

## 9. 最小可执行改动清单

1. 修改 `src/prompts.js` 中 `id: 'guide'` 的任务定义。
2. 修改 `src/app/promptConfig.js` 中 `GENERATION_RUNTIME_GUARDS.guide`。
3. 保持 UI 和存储结构不变。
4. 用 3 个脑洞验证：世情复仇、追妻倒计时、悬疑/脑洞类各一个。
5. 通过后再考虑是否为导语结果增加结构化拆解。
